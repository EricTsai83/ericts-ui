---
title: Designing Reduced Motion
description: How to respect reduced motion preferences without removing useful interface feedback.
---

Motion can make an interface easier to understand. A sidebar that slides in from
the left tells the user where it came from. A dialog that fades in helps
establish that something changed. Good animation gives shape to state changes.

But not every user wants that motion. Some users enable reduced motion at the
system level because movement, parallax, zooming, or large transitions can feel
distracting or physically uncomfortable. Supporting that preference is not just
an accessibility detail. It is part of designing motion responsibly.

Reduced motion does not mean removing every animation. A better mental model is:
remove the movement that creates visual burden, while keeping the feedback that
helps the interface feel clear.

## Reading the user's preference

You can read the user's preference with the `prefers-reduced-motion` media
query:

```tsx
"use client";

import * as React from "react";

const REDUCED_MOTION_QUERY = "(prefers-reduced-motion: reduce)";

export type ReducedMotionPreference = "system" | "reduce" | "no-preference";

export function useReducedMotion(
  preference: ReducedMotionPreference = "system",
): boolean {
  const [systemPrefersReducedMotion, setSystemPrefersReducedMotion] =
    React.useState(false);

  React.useEffect(() => {
    if (preference !== "system") {
      return;
    }

    const mediaQuery = window.matchMedia(REDUCED_MOTION_QUERY);

    const updatePreference = () => {
      setSystemPrefersReducedMotion(mediaQuery.matches);
    };

    updatePreference();

    mediaQuery.addEventListener("change", updatePreference);

    return () => {
      mediaQuery.removeEventListener("change", updatePreference);
    };
  }, [preference]);

  if (preference === "reduce") {
    return true;
  }

  if (preference === "no-preference") {
    return false;
  }

  return systemPrefersReducedMotion;
}
```

One important detail is that `window.matchMedia` is inside `useEffect`.

Even if this hook is only used in a Client Component, frameworks like Next.js may
still render the component on the server first before hydrating it in the
browser. That means browser-only APIs like `window` should not be read during
render. Putting `matchMedia` inside `useEffect` keeps it on the client.

The hook also calls `updatePreference()` immediately. Without that, the initial
state would always be `false`, even if the user already has reduced motion
enabled.

The optional `preference` argument lets a component use a local override:

```tsx
const shouldReduceMotion = useReducedMotion("reduce");
```

That is useful for demos, previews, settings panels, or design tools where you
want to show a reduced-motion mode without changing the entire website.

## Is a hook a component?

A hook is not a component. A component renders UI. A hook is reusable behavior
that a component can call.

That distinction is useful for a registry. `useReducedMotion` should not be
packaged as a visual component, because it does not own any markup. It should be
registered as a hook, then demonstrated through a component that uses it.

In this registry, the hook is installed to:

```txt
hooks/use-reduced-motion.ts
```

The reusable unit is the hook. The preview component is only there to show how
the hook changes animation behavior in a real interface.

## Reducing motion in a component

Here is a sidebar that normally slides out of view when closed:

```tsx
import { motion } from "motion/react";
import { useReducedMotion } from "@/hooks/use-reduced-motion";

export function Sidebar({ isOpen }: { isOpen: boolean }) {
  const shouldReduceMotion = useReducedMotion();
  const closedX = shouldReduceMotion ? 0 : "-100%";

  return (
    <motion.div
      animate={{
        opacity: isOpen ? 1 : 0,
        x: isOpen ? 0 : closedX,
      }}
      transition={{
        duration: shouldReduceMotion ? 0.12 : 0.24,
        ease: "easeOut",
      }}
    />
  );
}
```

When reduced motion is not enabled, the sidebar moves horizontally. That movement
can be useful because it communicates spatial direction.

When reduced motion is enabled, the sidebar no longer travels across the screen.
Instead, it stays in place and changes opacity. The user still receives visual
feedback, but they do not need to track a large moving surface.

That distinction matters. We are not deleting the interaction. We are changing
the type of feedback.

## Without Motion

You do not need Motion to respect reduced motion. The same idea can be
implemented with React state and CSS transitions:

```tsx
import { useReducedMotion } from "@/hooks/use-reduced-motion";

export function Sidebar({ isOpen }: { isOpen: boolean }) {
  const shouldReduceMotion = useReducedMotion();
  const closedX = shouldReduceMotion ? 0 : -320;

  return (
    <aside
      style={{
        opacity: isOpen ? 1 : 0,
        transform: `translateX(${isOpen ? 0 : closedX}px)`,
        transitionDuration: shouldReduceMotion ? "120ms" : "240ms",
        transitionProperty: "opacity, transform",
        transitionTimingFunction: "cubic-bezier(0.215, 0.61, 0.355, 1)",
      }}
    />
  );
}
```

This version has no animation dependency. Reduced motion is handled by changing
the distance, not by removing the state feedback completely.

You can also scope the preference to a single preview:

```tsx
import { useState } from "react";
import {
  useReducedMotion,
  type ReducedMotionPreference,
} from "@/hooks/use-reduced-motion";

export function SidebarPreview() {
  const [preference, setPreference] =
    useState<ReducedMotionPreference>("system");
  const shouldReduceMotion = useReducedMotion(preference);

  // Render local controls that update `preference`.
  // Only this preview changes behavior; the rest of the site is untouched.
}
```

## Using MotionConfig as a global safety net

If your application uses Motion throughout the UI, you can also set a global
reduced motion policy with `MotionConfig`:

```tsx
import { MotionConfig } from "motion/react";

export function AppProviders({ children }: { children: React.ReactNode }) {
  return (
    <MotionConfig reducedMotion="user">
      {children}
    </MotionConfig>
  );
}
```

When `reducedMotion` is set to `"user"`, Motion respects the user's system
preference. If the user has enabled reduced motion, Motion will avoid transform
and layout animations, while still allowing safer properties like `opacity` and
`backgroundColor`.

This is useful because the default is `"never"`, so Motion will not automatically
follow the user's preference unless you configure it.

Wrapping your application with `MotionConfig` gives you a baseline:

```tsx
<MotionConfig reducedMotion="user">
  {children}
</MotionConfig>
```

That way, you do not need to remember to handle reduced motion manually in every
single component.

Still, `MotionConfig` should be treated as a safety net, not a replacement for
design judgment. Some components deserve more deliberate reduced-motion
behavior. A sidebar, drawer, carousel, command menu, or page transition may need
a custom fallback because the motion is central to how the interaction feels.

## The design principle

A good reduced motion implementation asks:

- Is this motion helping the user understand what changed?
- Is the element moving across a large distance?
- Does the animation require the user to visually track movement?
- Could opacity, color, or an instant state change communicate the same thing
  with less burden?

For frequent product interactions, motion should be quiet. For users who prefer
reduced motion, it should be even quieter.

Reduced motion is not about making the interface lifeless. It is about
separating useful feedback from unnecessary movement. Keep the signal. Reduce the
motion.
